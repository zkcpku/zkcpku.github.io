git基本概念

commit命令

用户创建的检查点成为提交，每个提交代表了一个时间点的内容版本。commit命令是git中的一种组件。

git需要用户在每次创建提交时提供提交消息

git可以离线操作

commit命令实际上就是仓库中每个文件的快照。

日志(log) ——在使用git记录代码编写过程的前提下，使用git浏览历史

log 日志显示了提交列表，可以说提交commit是日志log的一部分。

  • 导航到代码的保存目录并输入git log

    结果将显示所有的commit内容，从最近的开始往下排列

    每个commit都有一个ID（类似于序列号，可以唯一的标识每个commit并使你能够引用它）、作者、日期和与之相关的注释信息（解释了自上次commit出现了哪些更改）

使用git diff命令——查看某个commit带来了那些更改

  • 输入git diff 不按回车
    • 后面：
      • 首先复制下第一个commit ID（旧的）
      • 然后复制第二个（新的）
      • 并将它们都粘贴到命令行中
      • 注意：如果输入提交 ID 的前四个或更多个字符更为轻松，则你可以这样做，而不必粘贴整个 ID。
    • 回车
    • ---旧的文件名 + 修改日期
    • +++新的文件名 + 修改日期
    • @@-旧的起始行数，旧的终止行数 +新的起始行数，新的终止行数@@
    • 下方的表示：
      • 黑色行代表没有更改
      • 带有减号的红色行表示被删除了
      • 带有加号的绿色行表示被增加的，即存在在第二个版本中的

git diff与FC或diff程序相似，但它不是对比两个文件，而是对比git中某个文件的不同版本

对于提交频率

  • 由于可以选择何时进行提交，因此，你可能想知道该多久提交一次更改。保持较小的提交通常是一个好做法。随着两个版本之间的差异越来越大，易于理解性和实用性都会越来越低。但是，你也不希望使提交过小。如果总是在每次更改一行代码后保存提交，则历史记录会因短时间内包含大量提交而变得难以读懂。

  • 为每项逻辑更改进行一次提交是很好的经验法则。例如，如果改正了一处打字错误，然后在文件的另一部分中改正了一个错误，则应为每项更改进行一次提交，因为这两项更改在逻辑上是独立的。如果这样做，每次提交都将具有一个易于理解的目的。Git 允许你在每次提交时都编写一条简短的信息来说明更改了什么。如果每次提交都包含一项逻辑更改，这条信息会更有用。

  • 选择何时提交是一种主观判断，而且不一定一成不变。在选择是否提交时，只需记住：每个提交都应具有一个清晰的逻辑目的，而且绝不应在不进行提交的情况下完成过多工作。

仓库(repository)——多文件git提交（追踪多个文件）

git仓库是文件的集合，是Git跟踪的聚合单元。
计算机上可以有多个不同的git仓库，仓库之间可以进行交互，commit提交可以在其对应的仓库内部进行交互。

使用多种语言进行编写代码,常见的方法是使用跨越多个文件的功能。
举个例子，当其中一个文件对另一个文件编写的函数进行调用时，如果要改变参数，则需要对两个文件中的相关内容同时改变。
所以追踪多个文件在html、css文件或者小说章节（比如修改主人公的姓名）中等等都是非常重要的。

在从事某个项目时，通常会有多个需要同时跟踪的文件，git将这些文件的集合称为“仓库”。
当你在git中保存文件的一个版本（commit），都将保存仓库中每个文件的版本。
如果你的仓库中只有一个文件，也是成立的。

另一方面，你可以在单次提交时，更改所有文件。

  • git log --stat
    该选项会给出每次提交，所更改的文件的统计数据。
    类似于git log，但每个提交也会显示该提交所影响的文件（即该次提交修改的文件）。
    绿色的加号表示新增，红色的减号表示删除。加号和减号的数量与更改的大小是成正比的。

  • 按Q停止查看git log输出

  • 输入git diff以及相比对的两次提交的ID，可以分别看到多个文件的修改状况。

git clone命令

用git下载的不只是实际的文件（也就是最近commit的文件），而是要下载整个历史记录文件。
也就是要将整个仓库从一台计算机搬到另一台计算机。
git clone可以创建仓库的克隆。

  • 要克隆版本库，请运行 git clone（后跟空格和版本库 URL）。(需要联网操作)

获得彩色输出

要获得彩色的 diff 输出，请运行 git config --global color.ui auto

git config命令只是更改git中的设置
--global选项表示将应用到所有的git项目

git 错误和警告解决方法

  • Should not be doing an octopus（不应执行 octopus）

    Octopus 是 Git 用来合并多个不同代码版本的一种策略。如果在不合适的情况下尝试使用该策略，则可能会出现此消息。

  • You are in 'detached HEAD' state（你处于“分离的 HEAD”状态）

    Git 将你目前所在的提交称为 HEAD。可通过切换到前一个提交来“分离”HEAD，这在下一个视频中有说明。虽然此警告听起来不太好，但分离 HEAD 实际上不是坏事。Git 只是向你发出警告，以便你知道自己正在这样做。

检出(git checkout)——检出代码的旧版本 恢复之前版本

在Git中，checkout 一个 commit 是指将所有文件重置到 进行commit时文件的状态。

复习Git command review

  • Compare two commits, printing each line that is present in one commit but not the other.

    git diff will do this. It takes two arguments - the two commit ids to compare.

  • Make a copy of an entire Git repository, including the history, onto your own computer.

    git clone will do this. It takes one argument - the url of the repository to copy.

  • Temporarily reset all files in a directory to their state at the time of a specific commit.

    git checkout will do this. It takes one argument - the commit ID to restore.

  • Show the commits made in this repository, starting with the most recent.

    git log will do this. It doesn't take any arguments.

git实战

.git文件夹

在项目的根目录下，有一个隐藏的文件夹.git，用来保存所有历史纪录。

  • 在输入ls可以看见所有文件
  • 在输入ls -a时还可以看见隐藏的文件

git init——创建仓库

  • 在项目所在的目录中打开git bash
  • 运行git init即初始化或创建新的git仓库，也就新建了.git文件夹
  • 注意：新建的Git中运行git log会发现，其中并没有commit，也就是说，新建的仓库中不含任何东西。我们必须自行commit提交
    • 这里有两个原因：
      • 我们可能不想提交目录中的所有内容
      • git如果自动提交的话，我们将不能写入自己的提交信息注释

Git 的版本库和目录

  • 每个 Git 版本库均关联到一个特定目录，即你在其中运行了 git init 的目录。版本库仅包含该目录及其子目录中的文件，而且不同的版本库可以位于不同的目录中。
  • 注意：你经常会看到，某个目录中的 Git 版本库仅包含或仅跟踪该目录中的某些，而不是所有文件。本课后面将介绍其工作原理。

git status——常用命令

通过该命令，我们可以看到最后提交后更改的文件，即显示出哪些内容未被跟踪，尚未commit。此外，该命令也会显示出最近的一次commit（若没有则为Initial commit）

git提交

  • 请切记，进行小提交，且在每次逻辑更改之后 进行一次提交是不错的想法。
  • git允许你明确指定要提交的更改。
    • git会使用一个称为存储区域的临时区域，每次可以将一个文件添加到存储区域中。
    • 在准备提交时，git会将存储区域中的所有内容绑定起来一起提交，然后添加到仓库中。
  • git add + 文件——将文件添加到暂存区
  • git reset + 文件——如果你意外地将某个文件添加到暂存区中，可以使用 git reset 删除它。此命令会从暂存区中删除该文件，但它仍在你的工作目录中
  • 再次运行git status可以看到显示出一些待提交更改的新文件new files
  • 修改完成后使用git commit，完成一次版本提交

    如何编写提交信息

    • 你将向你的反思版本库进行第一次提交。在你这样做时，需要编写提交信息，以说明所做的更改。在第 1 课快要结束时提供了适用于你的平台的“设置工作空间”视频，如果你遵循此视频中的说明，则在运行 git commit 时，所选的编辑器将会出现，让你可以编写提交信息。如果收到错误消息，则应尝试重温第 1 课中的说明，并确保正确设置了文本编辑器。
    • 你也可以通过命令行指定提交信息，即运行 git commit -m "Commit message"，而不是 git commit。设置编辑器仍然是一种好的做法，因为这可让你更轻松地编写较长的提交信息，以全面说明所做的更改。

git diff新用法——对比暂存区与最新版本

  • 单独使用git diff而不加参数时，可以对比暂存区与工作区，显示出更多的改变信息。
    常常与git status搭配使用，从而得到工作区中untracked的文件和unsave的文件信息。

  • git diff --staged 可以对比暂存区和最新版本，可以方便检查即将commit的版本和已存在最新版本之间的差别。

  • git reset --hard可以放弃当前工作区和暂存区的所有工作，相当于直接返回至最新版本。

    • ！！！千万小心！！！
      该步一旦执行无法恢复

  • 注意区分工作区、暂存区、版本仓库三者关系

    • 工作区working directory：文件现在所处的状态
    • 暂存区staging area：已经git add的文件，即将commit的文件
    • 版本仓库repository&commit：git log可以查询到ID的文件
  • 三者修改的新鲜程度：工作区>暂存区>版本仓库

提交ID的产生

提交ID与内部的修改、创建提交的作者、时间都有关，因而即使做了相同修改，提交ID也并不一样。

分支(branch)

  • 常用分支命名：

    • master分支：大多数git仓库中主要分支的名称，创建git仓库时自动创建master分支。它是你的产品质量分支，绝对不会崩溃，始终保持运转
    • development分支：用来包含正在进行工作的项目内容
    • 实验性分支：完全料到它们很大可能或者一定会崩溃，是你用来研究的内容，也是希望能和同事和朋友分享的内容
      • exp分支：实验性分支，用来扩展新功能前的beta版本
      • feature分支：类似于exp分支，用来实现特定功能
      • 与朋友协作时，也可以用branch分割出每个人的工作
    • bug-fix分支：可以用分支来进行bug的修改，最后合并，从而能够分割出这部分工作

注意：当checkout一些旧的commit时，git会提示detached head消息，即警告你 你所研究的这个commit没有用分支名称进行标记，即处于游离状态或位于master分支上。

  • 可以使用checkout检出分支，只是使用的是人类能看懂的分支名称而非ID号码
    • 当checkout到一个分支时，然后进行commit时，分支标记会自动更新到该commit上，也即在该分支上的commit
    • 该分支会一直保持检出状态，也即会一直停留在该分支上，而不会自动跳至master分支
  • 顶点(tip)：分支中的当前最后一个commit被称为该分支的顶点
  • 单个commit上可能会出现多个分支标记，但是更新 新的commit时只会更新当前已检出的分支，其他分支不受影响

git branch命令

  • git branch可以显示出已存在的分支，旁边的星号表示当前检出的分支
  • git branch new-branch-name可以创建新的branch，命名为new-branch-name

  • git checkout new-branch-name可检出new-branch-name分支并进行更改

  • 分支类似于上下文切换，可以保存从前的一切工作，在增加新功能等场合下使用可以让工作更有条理性。

git log命令

  • git log --graph --oneline ... ... ...等可以可视化分支组成
    • --graph以图形形式显示分支与commit连接方式，即绘制历史记录图表
    • --oneline单行显示commit信息，省去作者和时间信息
    • ...表示查看的分支，可以有多个分支，省略不写则只查看当前已检出的分支
      • git log以当前commit或任意指定分支的最新commit为起点，一直往前跟踪，直到到达initial commit（没有父commit的commit），即root。具体可参考父commit
    • 上面三个可以任意组合，也可以不写任何一个

  • git log --graph --decorate --oneline --simplify-by-decoration --all

    • 说明：
      • --decorate 标记会让git log显示每个commit的引用(如:分支、tag等)
      • --oneline 一行显示
      • --simplify-by-decoration 只显示被branch或tag引用的commit
      • --all 表示显示所有的branch，这里也可以选择，比如我指向显示分支ABC的关系，则将--all替换为branchA branchB branchC

父commit(parent)

  • commit的存储方式为单向链表，每个commit只会记录他的父commit，也即上一个commit。这样在运行git log时，只需从顶点中搜索到要查询的分支的顶点，然后向上遍历即可。
  • 由于仅保存父commit而不保存子commit，所以在checkout旧的commit时就可能会丢失后面的信息。
  • 可达性：注意，由于只记录一个父commit，因而在使用log进行搜索时，并不会将所有commit全部列举(比如并列分支上的commit以及子commit)

合并分支

git 可以将两个分支进行自动合并：

  • 相同的行会自动添加

  • 不同的行

    • 需要参考原始版本
    • 如果原始版本中有，但一个分支将其删除了，则最终没有
    • 如果原始版本无，但一个分支将其增加了，则最终有
    • 如果两个分支都对其中一处进行了修改，不管这两处修改是应该同时加入，还是应该加入其中一个，git都会检测出冲突并报错警告

  • 在分支合并中，原始版本即分支交叉（fork）处的最近一个commit文件

  • 特别注意：合并后的commit会同时存储两个父commit的提交信息，即该链表同时指向两个旧的commit，因此在使用git log时会显示两个分支上的commit（按时间顺序进行排列）
  • 新的commit会被附加到两个分支中的一个指定的分支上，这时我们即可以删除另一个分支。但要注意，删除分支只是删除了commit上的标签，但并未删除这些commit，事实上，我们仍可以通过git log访问到这些commit。更多可查看垃圾收集过程

具体操作：

  • 首先检出checkout希望被附加的分支，比如master分支
  • git merge master coins创建master分支和coins分支的合并版本。应当注意：由于是合并两个分支，因此在命令行中按何顺序键入分支并不重要。关键是要记住，git merge 始终将所有指定的分支合并到当前检出的分支中，并为该分支新建一个提交。
  • 填写注释信息，注释中自动补充了合并分支的信息
  • 可以用git log --graph --oneline来观察确定合并结果
  • 应当注意到：如果此时仅仅使用git log，则只会按时间顺序显示出commit。如果希望对比历史commit中相邻的修改，很难用git diff确定哪两个ID相邻。
  • 我们可以采用git show commit_ID的命令将commit与所在分支的commit进行对比

  • 删除分支

    • 合并分支后，我们可以对分支进行删除：
    • git branch -d branch_name命令可以删除分支标签，但不会删除commit
    • 其中 -d表示删除
    • 注意到：合并后我们依然能够访问到被删除分支上的commit，但当我们未合并直接删除分支后，我们将很难在访问到那些commit，git会定期自动垃圾回收。那意味着我们将放弃所有commit。有关内容可参看垃圾收集过程

对git merge的更多说明

  • git merge 还将在合并的版本中包含当前检出的分支。因此，如果检出了 branch1，并且运行 git merge branch2 branch3，则合并的版本会将 branch1 以及 branch2 和 branch3 组合起来。由于在你进行合并提交后 branch1 标签将会更新，因此，你不想将 branch1 中的更改包含在合并中是不可能的。有鉴于此，在合并之前应始终检出你打算合并的两个分支之一。应检出哪个分支取决于你想让哪个分支标签指向新的提交。

  • 由于检出的分支始终包含在合并中，因此，你可能已猜到，在合并两个分支时，无需在命令行上将两者都指定为 git merge 的参数。如果想将 branch2 合并到 branch1 中，只需键入 git checkout branch1，然后键入 git merge branch2 即可。键入 git merge branch1 branch2 的唯一原因是，它能帮助你对要合并的分支更加心中有数。

  • 另外，由于是合并两个分支，因此在命令行中按何顺序键入分支并不重要。关键是要记住，git merge 始终将所有指定的分支合并到当前检出的分支中，并为该分支新建一个提交。

合并冲突（Windows 与 Unix 系统之间的换行符）

  • 背景：按下键盘上的“Enter”键时，实际上是告诉计算机在文本文件中插入一个不可见的字符，以便指示计算机应新起一行。Unix 系统会添加一个名为“换行”符的字符（或者 LF 或 \），而 Windows 系统会添加两个字符，即“回车”和“换行”（或者 CRLF 或 \r\n）。

  • 老师 Caroline 的文件包含 LF，因为她的文件是在使用 LF 的 Mac OSX 上编辑的。如果某 Windows 用户要编辑 Caroline 的文件，Windows 文本编辑器可能会将所有 LF 均转换为 CRLF，以便该用户能够编辑文件。在该 Windows 用户将她的文件与 Caroline 的文件合并时，不同的 LF 和 CRLF 字符将会造成合并冲突。

  • 要修复此问题，Windows 用户应将 autocrlf 全局属性设置为 true：git config --global core.autocrlf true。有关更多信息，请访问以下网址：https://help.github.com/articles/dealing-with-line-endings/#platform-all

垃圾收集过程

如果删除某个分支并导致无法从现有的分支访问一些提交，则在 Git 的垃圾收集过程运行之前，仍可继续通过提交 ID 访问这些提交。除非你主动关闭此过程，否则它将不时自动运行。也可以通过命令 git gc 手动运行此过程。

远程仓库

  • 典型仓库：

    • 我们自己线上的仓库常被称为origin
    • 如果我们线上的仓库是fork他人的，通常还会有他人的原始仓库，被称为upstream
      • 在我们希望能和upstream仓库进行沟通时，我们需要拉取请求。这中间可能需要解决更复杂的冲突，因此我们需要常常fork该仓库以保持最新，才能在本地进行合并（注意：github线上不能解决冲突问题）
      • 因此，初次时将upstream仓库加入，之后常常执行git pull upstream master（或其他分支）是应当选择的方法

  • git remote add origin ......表示仓库地址，该命令可以将云端的仓库命名为origin（当目录中只有一个仓库时，通常的做法是将其命名为origin）

  • git remote可以显示出已存在的在线仓库
  • git remote -v -v表示显示详细信息，将显示获取数据和推送数据的URL,大多数情况下两个URL相同
  • git push 远程仓库名 将要提交本地的分支名向远程仓库发送更改
  • git pull 远程仓库名 将要同步远程的分支名向远程仓库获取数据
  • 应当注意，如果我们是克隆线上的仓库，git会自动remote链接到远程仓库，不需要更多的配置
  • git fetch当远程仓库发生更改，本地仓库也发生更改，可以使用该命令更新远程分支的本地副本，而实际的本地版本保持不动。由于两个更改可能存在冲突，所以用这种命令来更新是更好的选择。即可以理解成出现了origin/master的新分支来保存线上的更改，而master分支依然为本地的更改。之后可以用git merge合并本地master分支。事实上，上面叙述的正是git pull的详细操作过程，即git pull=git fetch+git merge。具体操作：
    • git fetch origin会更新origin远程仓库中的所有分支的本地副本
    • 接着可以运行git log origin/master检查本地副本
    • 也可以运行git diff origin/master master
    • 注意：github执行快速合并，即如果要合并的两个commit是父子关系或者祖先关系，即二者可通过链表的箭头（不管中间差了几个）连接，则在合并的时候只会在branch上更新commit，而不会出现两分支合并的情况。

fork

  • github提供fork，可以直接在服务器上复制他人的代码，然后我们可以在自己的代码上进行实验更改，而不会污染到原作者
  • fork类似于clone，只不过基于线上
    • clone可以将线上的仓库克隆到线下，也可以将线下某个目录中的仓库克隆到线下另一目录
  • fork可以更便捷地向原作者反馈建议
